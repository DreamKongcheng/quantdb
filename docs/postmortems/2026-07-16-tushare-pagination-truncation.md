# Tushare 分页静默截断复盘

- 日期：2026-07-16
- 影响数据集：`tushare.index_member_all`
- 严重程度：高
- 状态：已修复，待刷新正式数据库中的受影响分区

## 摘要

`index_member_all` 首次接入时将客户端请求页长配置为 4,000，但 Tushare 服务端
实际单页最多只返回 3,000 行。通用分页器使用“返回行数小于请求页长”作为最后一页
判据，因此首屏返回 3,000 行后被误判为查询完成，并将不完整结果以 `SUCCESS` 状态
原子提交。

真实数据共有 5,864 行：第一页 3,000 行，第二页 2,864 行。受影响的正式分区只保存
了前 3,000 行，导致部分股票缺少申万行业归属。

## 发现经过

用户在 `quantdb update` 进度中注意到：

```text
tushare.index_member_all ... 数据行=3000
```

3,000 恰好是常见服务端单页上限。随后直接使用 Tushare 的 `limit` / `offset` 参数验证：

| offset | 返回行数 |
| ---: | ---: |
| 0 | 3,000 |
| 3,000 | 2,864 |
| 5,864 | 0 |

合计 5,864 行，注册主键 `(ts_code, l3_code, in_date)` 无重复。

## 根因

原分页终止条件是：

```python
if len(page) < requested_page_size:
    break
```

这个条件隐含了一个未经验证的假设：服务端会严格遵守客户端传入的 `limit`。Tushare
部分接口会静默将 `limit` 截断为各自的服务端上限，因此“短页”既可能表示最后一页，
也可能只是服务端上限小于请求值。

本次还有三个促成因素：

1. 接口页长依据记忆设置，没有逐项引用当前官方文档并做真实多页验证。
2. 测试只确认请求中携带了配置的 `limit`，没有模拟“服务端上限小于请求页长且仍有
   下一页”的行为。
3. 首次真实冒烟只验证了接口可调用，没有在交付前对全量行数和第二页进行核验。

## 修复

### 通用机制

分页器不再根据短页结束，而是始终按实际返回行数推进 `offset`，直到服务端明确返回
带完整字段的空页。这样即使服务端上限小于请求页长，也不会静默截断。

保留了两项失败保护：

- 服务端忽略 `offset` 并重复第一页时，检测首行哈希不推进并拒绝提交。
- 后续页发生网络、字段或 SDK 异常时，整个逻辑分区失败，不提交已经获取的部分页。

该策略会为每次完整查询增加一次空页请求。这是有意识的完整性优先取舍。

Tushare 原始 HTTP 响应含有 `has_more`，理论上可以无额外请求地判断结束；但当前使用
的 Tushare SDK 1.4.29 在 `DataApi.query()` 中只返回 DataFrame，没有公开分页元数据。
项目不读取 SDK 私有 token、URL 等属性。若后续 SDK 公开 `has_more`，可以替换空页
探测，在保持完整性的同时消除额外请求。

### 接口配置

- `index_member_all.page_size` 按官方文档改为 2,000。当前服务端实测允许 3,000，
  但客户端遵循较保守的公开契约。
- `stk_limit.page_size` 从通用 6,000 改为官方上限 5,800。

### 回归测试

新增测试覆盖：

- 请求页长为 4，但服务端每页最多返回 3 行时，仍能获取全部 5 行。
- `index_member_all` 按 2,000 / 2,000 / 1,864 / 0 完成全部 5,864 行分页。
- 服务端忽略 `offset` 时显式失败。
- 尾页为空前发生网络或无字段响应时，不返回部分数据。

## 全接口审计

审计范围为当前注册的 14 个 Tushare 接口。官方未注明上限时，表中保留“未注明”，
并以真实服务端分页探测补充判断。

真实探测参数如下：

- 日分区接口统一使用 `trade_date=20250715`。
- `trade_cal` 使用 `exchange=SSE,start_date=20250101,end_date=20251231`。
- `index_weight` 使用
  `index_code=000985.CSI,start_date=20250601,end_date=20250630`。
- `stock_basic` 分别查询 `list_status=L/D/P`，`index_basic` 逐一查询
  `market=MSCI/CSI/SSE/SZSE/CICC/SW/OTH`，`index_classify` 使用 `src=SW2021`。
- `namechange` 和 `index_member_all` 不传过滤条件。通用探针先请求
  `limit=6000,offset=0`，再按实际返回行数推进 `offset`；有专用页长的接口另按注册配置
  复测。

| 接口 | 官方单次上限 | 修复后页长 | 真实分页结果或结论 |
| --- | ---: | ---: | --- |
| `stock_basic` | 6,000 | 6,000 | L/D/P 为 5,529 / 337 / 0，下一页为空 |
| `namechange` | 未注明 | 6,000 | 6,000 + 6,000 + 2,136 |
| `trade_cal` | 未注明 | 6,000 | 单自然年 365 行 |
| `daily` | 6,000 | 6,000 | 样本日 5,408 行，下一页为空 |
| `adj_factor` | 未注明 | 6,000 | 样本日 5,445 行，下一页为空 |
| `daily_basic` | 6,000 | 6,000 | 样本日 5,408 行，下一页为空 |
| `suspend_d` | 未注明 | 6,000 | 样本日 13 行 |
| `stk_limit` | 5,800 | 5,800 | 样本日总计 7,186 行，跨页主键无重复 |
| `index_basic` | 未注明 | 5,000 | CSI 为 5,000 + 3,802 |
| `index_daily` | 未注明 | 6,000 | 样本日 6,000 + 4,923，跨页主键无重复 |
| `index_dailybasic` | 3,000 | 3,000 | 样本日 11 行 |
| `index_weight` | 未注明 | 1,000 | 中证全指样本月 1,000 x 5 + 74 |
| `index_classify` | 未注明 | 6,000 | 511 行 |
| `index_member_all` | 2,000 | 2,000 | 总计 5,864 行，跨页主键无重复 |

本次真实审计只在 `index_member_all` 发现已发生的静默截断。`stk_limit` 的旧配置与
官方契约不符，但当前服务端实测仍返回了完整分页；现已按官方上限收紧。其余接口未
发现同型缺失，通用空页终止机制也覆盖了官方未注明上限或未来上限变化的风险。

`index_member_all` 的 5,864 行还核验了整行重复、注册主键重复和跨页主键交集，三项
结果均为 0。

另有一项独立契约风险：`index_daily` 官方文档仍将 `ts_code` 标为必填，当前服务端
实测支持只传 `trade_date` 获取全量指数。本项目为覆盖境内外指数而使用该行为；如果
上游收紧，需要改为按指数代码同步。这不是本次分页事故，但继续记录为已知风险。

提交前完整性审查还发现：维护命令原先默认同步到今天。盘中执行时，日线可能仍是
半成品；允许空结果的 `index_daily` 还可能把尚未发布的数据记录为成功空分区，后续
增量更新会直接跳过。现已将 `update()`、`update_indices()` 和 `health()` 的默认截止日
统一改为昨天；调用者仍可显式传入今天，但需要自行承担当日数据尚未完整的风险。

## 数据修复

正在运行的旧进程不会加载本次代码修改。待其退出并释放数据库锁后，执行：

```bash
uv run quantdb sync tushare.index_member_all --refresh
```

预期结果为 5,864 行。`refresh` 会在获取、分页和完整性校验全部成功后原子替换旧的
3,000 行；中途失败不会破坏已有数据。

由于 `stk_limit` 是交易日分区，如果需要用新分页机制重新验证已有历史分区，应对
目标日期范围显式执行 `--refresh`。本次真实审计的 7,186 行样本已经确认旧逻辑获取
完整，因此没有证据表明现有 `stk_limit` 数据已经缺失。

## 防再发规则

后续新增 Tushare 接口必须同时满足：

1. 在注册表中记录官方文档明确给出的单次上限，并使 `page_size` 不高于该上限。
2. 分页完成只能以空页为准，不能以短页为准。
3. 至少提供一个“服务端上限小于请求页长”的通用分页测试。
4. 可能超过单页的数据集必须做真实第二页探测，并核验主键跨页无重复。
5. 交付前记录全量行数；结果恰好等于 1,000、2,000、3,000、5,000 等边界值时，
   必须继续检查下一页。
6. 如果接口不支持或忽略 `offset`，不得复用通用分页器，应实现专用分区策略。
7. 增量维护默认只处理已结束日期；同步今天必须由调用者显式指定，并视为可重刷数据。
